GitHub API Optimization Plan

UX を落とさずに GitHub API を減らすための整理

現状の Desktop 側ロジックは「active workspace だけを同期する」という大きな方向性は良いが、 5 秒ループの中に重い処理が入りすぎている。特に PR status と preview deployment が同じホットパスに乗っており、 comments 同期も内部で status 解決を再実行している。問題はポーリングそのものではなく、 「高頻度で回す対象の切り分けが甘いこと」にある。

5 秒 現在の PR status キャッシュ TTL / Sync interval
20 秒 現在の PR comments キャッシュ TTL / Sync interval
60 秒 VS Code 拡張の webview 既定 refresh 間隔。非アクティブ時はさらに鈍い

先に結論

最適化の中心は GitHubSyncService で合っている。ただし、 SyncService を削るのではなく「scheduler に徹させる」方向で設計し直すのが良い。 具体的には status を 1 本の重い取得として扱うのをやめ、 PR identity / checks summary / comments / preview URL / overview のレーンに分けるべき。

active workspace only は維持 visible UI 優先 cheap sentinel 先行 重い fetch は lazy / 長 TTL ユーザー操作後だけ短周期

現状ロジック

現在の構成は「backend cache を温めて、frontend はそのキャッシュを読む」方式。 方向性は悪くないが、温める粒度と頻度に無駄がある。

Active Workspace

KeepAliveWorkspaces が active な workspace を backend に通知し、 GitHubSyncService はその workspace だけを対象にする。

良い点 全 workspace 一斉 poll ではない
5 秒ループ

refreshGitHubPRStatus

  • repo context 解決
  • 現在 branch / HEAD の解決
  • getPRForBranch による PR 特定
  • branchExistsOnRemote の確認
  • fetchPreviewDeploymentUrl による preview 探索

問題はこの 1 本に責務が集まりすぎていること。特に preview 探索は GitHub API コール数が膨らみやすく、5 秒ループに載せるべきではない。

20 秒ループ

refreshGitHubPRComments

  • review threads を GraphQL で取得
  • issue comments を REST で取得
  • ページネーション次第で API 呼び出し数が増える

現在の SyncService パスでは comments 同期の前に resolvePullRequestCommentsTarget が走り、 内部で fetchGitHubPRStatus を再実行する。 つまり comments ループが status ループを暗黙に二重化している。

UI 追加ポーリング
  • Actions / logs 表示中は workflow jobs と job logs が 3 秒ポーリング
  • RepositoryOverview は active 時に 60 秒 stale で再取得
  • hover や list item は lazy fetch で、ここは比較的まとも

現状の良い点と悪い点

良い点

  • active workspace のみに同期対象を絞れている
  • frontend は GitHub API を直接連打せず backend cache を見ている
  • workspace row / hover card は lazy で、常時 warm していない
  • review panel の一部 query は popover open 時のみで比較的軽い

悪い点

  • 5 秒で回す status の中身が重すぎる
  • preview URL 探索が hot path に直結している
  • comments 同期が status 解決を再実行している
  • interval が固定で、状態に応じた強弱がない
  • 1 つの status object に責務が集約されすぎている

主なボトルネック

Hot Path

1. Preview Deployment 探索

deployments 一覧取得に加えて、候補ごとに deployment statuses を確認するため、 条件次第で GitHub API の呼び出し数が大きく増える。

Redundant

2. Comments ループの二重 status 解決

comments 同期の前段で PR target を解決する際、 status 取得を再度走らせている。UX に寄与しない重複コスト。

Scheduler

3. 固定 5 秒

check 実行中でも idle 状態でも同じ頻度で回る。 状況に応じて interval を変える仕組みがない。

4. Status の責務過多

branch, PR identity, checks, preview, comments の関心が混ざっており、 1 つの再取得で全部を巻き込んでいる。

5. Overview / Actions の切り分け不足

Repository overview や workflow logs は本来「そのパネルが見えている時だけ」 強く同期すればよい。常時 warm 対象とは分けて考えるべき。

VS Code から持ち込める原則

VS Code の GitHub Pull Requests 拡張は、少なくとも現状の設計よりかなり慎重。 常時短周期でフル query を回すのではなく、軽い変化検知を先に挟む。

観点 現状 VS Code の原則 持ち込むべき判断
基本 refresh status 5 秒 / comments 20 秒 webview は既定 60 秒、非アクティブ時はさらに鈍い 平常時は 60 秒寄り、短周期は例外扱いにする
重い query の前処理 即フル取得 lastUpdateTime や maxKnownPR で先に変化検知 cheap sentinel を先に確認してから重い fetch に進む
非アクティブ時 active workspace 基準のみ window / tab 状態で interval を落とす visibility を scheduler 条件に含める
通知 polling 固定値中心 x-poll-interval を利用 取れる所は server hint を尊重する
PR list refresh 重い overview fetch に寄りやすい 新しい PR があるかだけ先に確認 一覧系は lightweight sentinel を持つ

提案アーキテクチャ

中央の考え方は「1 本の status を warm する」から 「複数レーンを scheduler が必要に応じて warm する」への移行。

Lane 分割

  • PR identity lane: PR 番号、URL、draft、mergeable など
  • Checks lane: CI 状態、最新 run、pending かどうか
  • Comments lane: review threads と issue comments
  • Preview lane: preview URL。完全に別扱い
  • Overview lane: labels, assignees, workflow list など

Scheduler 状態

  • Active / idle: 平常時。60 秒前後
  • Active / checks pending: CI 実行中。10〜15 秒
  • User action just happened: manual refresh, rerun, reviewer 更新直後のみ短周期
  • Inactive / hidden: 停止または 5 分程度
重要なのは active workspace だけでなく active surface も見ること。 例: review パネルを見ていないのに comments を高頻度で温める必要は薄い。

各レーンの推奨 TTL / トリガー

レーン 推奨間隔 主トリガー 狙い
PR identity active 60 秒 / inactive 5 分 branch 変更、HEAD 変更、manual refresh PR 番号や URL がそう頻繁に変わらない前提で軽く保つ
Checks pending 10〜15 秒 / idle 60 秒 workflow run 中、rerun 直後 CI 状態だけは比較的リアルタイムに維持
Comments review 可視時 30〜60 秒 review タブ表示、manual refresh 可視時は追従しつつ、背景では抑制
Preview 5〜15 分、または lazy PR 生成直後、head SHA 変更時、ユーザーが必要にした時 重い探索をホットパスから外す
Overview 5〜10 分 popover open、mutation 後 invalidate 一覧系の無駄 fetch を削る
Action logs 可視 + pending 時のみ 3 秒 pane open、run pending UX を維持しながら対象範囲を限定する

cheap sentinel の設計

Workspace sentinel

ローカルの headSha、current branch、worktree 状態。 ここが変わっていなければ PR identity の再解決頻度を落とせる。

PR sentinel

直前に解決した prNumber と updatedAt。 comments や overview の重い再取得前にまず軽い更新時刻を見る。

Checks sentinel

最新 run の state だけ先に見る。pending の間だけ短周期にし、 settled したら interval を落とす。

最小変更で効く順

Phase 1

  • previewUrl を status ループから外す
  • SyncService comments fetch に既知の PR target を渡す
  • 平常時 status interval を 5 秒から 30〜60 秒へ引き上げる

Phase 2

  • status を lane に分割する
  • cheap sentinel を導入して重い取得を gate する
  • comments を review 可視時中心に寄せる

Phase 3

  • overview / actions を共通 scheduler で統合管理
  • gh 呼び出し回数と rate header を可視化する
  • low-API mode を常設オプションとして追加する

UX への影響

維持できるもの

  • active workspace の PR 状態は見た目上ほぼ維持できる
  • CI pending 中の即時感は 10〜15 秒ポーリングで十分保てる
  • Actions / logs は可視時だけ今の速さを維持できる

少し遅くしてよいもの

  • preview URL
  • review comments の背景同期
  • assignees / labels / workflow list の overview 情報
UX を守る鍵は「全部を速くする」ではなく、「人間が見ているものだけ速くする」こと。 逆に言えば、見えていない preview や overview を 5 秒で温め続けるのは UX にほぼ寄与しない。

期待できる効果

  • secondary rate limit に触る頻度を大きく下げられる可能性が高い
  • active workspace 1 個あたりの steady-state API 使用量をかなり削れる
  • ボトルネックがレーンごとに見えるようになり、将来の調整がしやすくなる
  • 「何が UX 上本当に重要か」がコード構造にも反映される
